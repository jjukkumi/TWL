# 소프트웨어 설계
## 화면 설계
### **UI표준 및 지침**
#### 한국형 웹 콘텐츠 접근성 지침(KWCAG)
- 장애인과 비 장애인이 동등하게 접근할 수 있는 웹 콘텐츠 제작의 방법 제시
1. 인식의 용이성
    - 대체 텍스트 제공해야 함
    - 멀티미디어 대체 수단을 제공해야함
    - 콘텐츠를 명료하게 전달해야 함
2. 운용의 용이성
    - 키보드만으로도 접근할 수 있어야 함
    - 충분한 시간을 제공해야 함
    - 광과민성 발작 예방
    - 콘텐츠를 쉽고 편리하게 내비게이션 할 수 있어야 함
3. 이해의 용이성
    - 읽고 이해하기 쉬워야 함
    - 기능과 실행결과는 예측 가능해야 함
    - 선형 구조로 작성되고 논리적인 순서를 제공해야 함
    - 입력 오류를 방지하거나 정정할 수 있어야 함
4. 견고성
    - 마크업 언어의 문법을 준수해야 함
    - 접근성이 있어야 함
#### 전자정보 웹 표준 준수 지침
- 정부 기관의 홈페이지 구축 시 반영해야 할 최소한의 규약
    - 내용의 **문법** 준수
    - 내용과 표현의 분리
    - 동작의 기술 중립성 보장
    - 플러그인의 **호환성**
    - 콘텐츠의 **보편적 표현**
    - 운영체제에 독립적인 콘텐츠 제공
    - 부가 기능의 호환성 확보
    - 다양한 프로그램 제공

### **UI 설계 도구**
- 설계도는 결과물에 대해 미리보기 기능을 함
#### 와이어프레임
- 기획 초기 단계에서 제작하는 것으로, 페이지에 대한 대략적인 레이아웃이나 UI요소 등에 대한 뼈대를 설계
- 와이어프레임 툴 : 손그림, 파워포인트, 키노트, 스케치, 일러스트, 포토샵 등
#### 목업
- 와이어프레임보다 좀 더 실제 화면과 유사하게 만드는 정적인 형태의 모형
#### 스토리보드
- 와이어프레임에 콘텐츠에 대한 설명이나 페이지 간 이동 흐름 등을 추가한 문서
- 디자이너와 개발자가 최종적으로 참고하는 작업 지침서
- 서비스 구축을 위한 모든 정보가 담겨 있어야 함
- 스토리보드 툴 : 파워포인트, 키노트, 스케치 등
#### 프로토타입
- 와이어프레임이나 스토리보드 등에 인터랙션을 적용해 실제 구현된 것처럼 테스트가 가능한 **동적**인 형태의 모형
- 작성 방법에 따라 페이퍼/디지털 프로토타입으로 나뉨
- 프로토타입 툴 : HTML/CSS, 카카오 오븐 등
#### 유스케이스
- 사용자 요구사항(기능적)을 다이어그램 형식으로 문서화
- 각각의 유스케이스에 대한 명세서 작성

### **UI요구사항 확인**
- 새로 개발할 시스템에 적용할 UI관련 요구사항을 조사해 작성하는 단계
#### 1. 목표정의
- 사용자를 대상으로 인터뷰를 하고 사용자의 의견이 수렴된 비즈니스 요구사항을 정의
- 인터뷰 진행 시 유의사항
    - 사업적, 기술적 요구사항을 명확히 이해
    - 가능한 개별적인 진행
    - 한 시간을 넘기지 않는게 좋음
    - 사용자 리서치를 시작하기 전에 해야함
#### 2. 활동사항 정의
- 목표, 예산, 계획, 우선순위 지정
- 조사한 요구사항을 토대로 앞으로 해야 할 활동사항을 정의
- 기술의 발전 가능성을 파악하고 UI 디자인의 방향 제시

#### UI 요구사항 작성
- 여러 경로로 수집된 사용자의 요구사항을 검토하고 분석하여 UI 개발 목적에 맞게 작성해야 함
- 실 사용자 중심으로 작성
- 여러 사람의 인터뷰를 통해 다양한 의견을 수렴하여 작성
##### 3. 요구사항 요소 확인
- 파악된 요구사항 요소의 종류와 각각의 표현 방식 등을 검토
- 요구사항 요소 
    - 데이터 요구
        - 가장 먼저 확인
        - 객체간 특성을 기반으로 정리
    - 기능 요구
        - 무엇을 실행하는지
        - 동사형으로 정리
    - 제품/서비스의 품질
        - 품질, 서비스, 감성적인 부분
    - 제약사항
        - 개발 비용과 데드라인, 규제사항 등
##### 4. 정황 시나리오 작성
- 사용자의 요구사항을 도출하기 위해 작성
- 사용자가 목표를 달성하기 위해 수행하는 방법을 순차적으로 묘사
- 개발하는 서비스의 모습을 상상하는 첫번째 단계
- **사용자 관점**에서 시나리오 작성해야 함
##### 5. 요구사항 작성
- 정황 시나리오를 토대로 작성

### **품질 요구사항**
#### 기능성
- `적절한 기능이 정확하게 + 호환, 보안`
- 적합/적절성 : 지정된 작업, 목표한 바에 적절한 기능을 제공하는 능력
- 정밀/정확성 : 사용자의 요구대로 결과를 산출하는 능력
- 상호운용성 : 다른 시스템과 서로 어울려 작업할 수 있는 능력
- 보안성 : 정보의 접근 권한을 허용하거나 차단할 수 있는 능력
- 호환성 : 기능과 관련된 표준, 관례, 규정을 준수할 수 있는 능력
#### 신뢰성
- `결함이 나도 문제없어!`
- 성숙성 : 결함으로 인한 고장을 피해갈 수 있는 능력 (웬만해선 고장이 안나는 능력)
- 고장 허용성 : 결함 시에도 규정된 성능 수준을 유지하는 능력
- 회복성 : 고장 시 다시 회복하고 영향을 받은 데이터를 복구하는 능력
#### 사용성
- `얼마나 쉽고 편한지`
- 이해성 : 적합성, 사용 방법 등을 사용자가 이해할 수 있는 능력
- 학습성 : 어플리케이션을 학습할 수 있도록 하는 능력
- 운용성 : 운용, 제어할 수 있도록 하는 능력
- 친밀성 : 다시 사용하고 싶어하도록 하는 능력
#### 효율성
- `한정된 시간, 자원으로 많은 일처리`
- 시간 효율성 : 특정 기능 수행 시 적절한 반응 시간 및 처리 시간, 처리율을 제공할 수 있는 능력
- 자원 효율성 : 특정 기능 수행 시 적절한 자원의 양과 종류를 제공하는 능력
#### 유지보수성
- `개선 및 확장`
- 분석성 : 결함이나 고장의 원인 등을 식별 가능하게 하는 능력
- 변경성 : 결함 제거 또는 환경 변화로 인한 수정 등을 쉽게 구현하는 능력
- 안정성 : 변경으로 인한 예상 못한 결과를 최소화하는 능력
- 시험성 : 변경이 검증될 수 있는 능력
#### 이식성
- `다른 환경에서 적응`
- 적용성 : 다른 환경에서도 적절하게 사용될 수 있는 능력
- 설치성 : 임의의 환경에 소프트웨어를 설치할 수 있는 능력
- 대체성 : 다른 소프트웨어를 대신하여 사용될 수 있는 능력
- 공존성 : 다른 소프트웨어와 데이터 공유하며 공존할 수 있는 능력

### **UI 프로토타입 제작 및 검토**
#### 개요
- 사용자 요구사항을 기반으로한 동적인 형태(실제로 동작하는 것 같음)의 테스트 모델
- 필수 기능을 포함해야함
- 최대한 간단하게 만들어야함
- 실제 사용자를 대상으로 테스트 해야함

#### UI 프로토타입의 장단점
##### 장점
- 사용자를 설득/이해시키기 쉬움
- 요구사항을 점검하며 혼선을 예방하므로써 개발 시간 단축
- 사전 오류 발견 가능
##### 단점
- 프로토타입 제작으로 인해 작업 시간을 증가시킬 수 있음
- 자원 소모
- 부분적으로 작업 시 중요한 작업의 생략가능성

#### 프로토타이핑의 종류
##### 페이퍼 프로토타입
- 스케치등의 아날로그 방식으로 작성
- 저렴, 즉시, 간단
##### 디지털 프로토타입
- 완성 제품과 비슷하게 만들어야할 때
- 테스트/수정이 용이

#### UI 프로토타입 계획 및 작성 시 고려사항
##### 계획
- 목적-환경-핵심UI-인원-검증-해결-가이드
##### 작성
- 범위확인-목표확인-자원확인

#### UI 프로토타입 제작 단계
1. 요구사항 분석
- 사용자의 요구사항을 분석
- 기본적인 요구사항이 확정될 때까지 수행
2. 요구사항 작성(프로토타입)
- 실제 도구를 이용하여 작성
- 핵심적인 기능을 중심으로 개발
3. 테스트 및 피드백
- 사용자가 요구사항을 잘 수행하는지 확인
- 다양한 추가 및 수정 의견을 제안할 수 있음
4. 수정 및 승인 (사용자가 승인할 때 까지 3-4단계 반복)
- 작성된 프로토 타입을 기반으로 수정과 합의
- 개발자는 사용자가 요청한 제안 사항을 수용하여 보완 작업

### **UI 설계서 작성**
#### UI 설계서의 개요
- 사용자의 요구사항을 바탕으로 UI 설계를 구체화하여 작성하는 문서
#### UI 설계서 작성 순서
1. 표지 작성
- 다른 문서와 혼동되지 않도록 시스템명 또는 프로젝트 명을 포함
2. 개정 이력 작성
- UI 설계서가 수정될 때 마다 어떤 내용이 수정되었는지를 정리한 문서
- 초안 작성 이후 변경될 때 마다 버전 0.1씩 증가
3. 요구사항 정의서 작성
- 사용자의 요구사항을 확인하고 정리한 문서
- 요구사항 별 적용여부 기록
4. 시스템 구조 작성
- UI 요구사항 및 프로토타입에 기초해 전체 시스템(사용자, 관리자 등)의 구조를 설계
5. 사이트 맵 작성
- 시스템 구조를 바탕으로 사이트에 표시할 콘텐츠를 메뉴별로 구분하여 설계
6. 프로세서 정의서 작성
- 사용자 관점에서 사용자가 요구하는 프로세스를 작업 순서에 따라 작성하여 UI 전체적인 흐름을 파악할 수 있음
7. 화면 설계
- UI 프로토타입 및 프로세스를 참고하여 필요한 화면을 페이지별로 설계
- 각 화면별로 표지 + 스토리보드
